# HyDE — Hypothetical Document Embeddings (P3.3)

Flag-gated retrieval optimization that replaces the raw-question
embedding with an averaged embedding of one or more synthetic "excerpt"
answers generated by the chat model.

Reference: Gao, Ma, Lin, Callan 2022, *"Precise Zero-Shot Dense Retrieval
without Relevance Labels"* (arXiv:2212.10496).

## Why

A user asks: *"What's our refund policy for B2B customers?"*

The embedding of that **question** is closest to other **questions** about
refunds — not to the **declarative statements** in our actual policy docs:

> "Refunds for enterprise-tier customers are issued within 30 days of a
> written request to billing@..."

Distance between question-phrasing and statement-phrasing is real. On a
year-long KB where the answer lives in a reference-style document, the
raw-question embedding misses the right chunk — it ranks ChatGPT-ish
follow-ups and Stack Overflow-style posts higher than the actual policy.

HyDE sidesteps the mismatch by asking the chat model to write a
plausible answer *as if it were quoting the document*, then retrieves on
that synthetic statement's embedding. Retrieval snaps to the real doc
chunk because the two embeddings now live in the same phrasing-space.

## When it helps

- **Abstract queries**: "How do we handle X?", "What's our approach to Y?"
- **Definitional queries**: "What is our Z policy?"
- **Long, year-old KBs**: where question and answer phrasings have
  drifted apart
- **Lexical mismatches**: query uses modern terminology, docs use legacy
  terminology (or vice versa)

## When it doesn't help

- **Keyword-style queries**: "error code 503", "s3_region config key" —
  the lexical overlap is already strong; HyDE adds latency without win.
- **Fresh, well-indexed FAQs**: a 50-doc FAQ where hybrid retrieval
  already finds the right chunk.
- **Queries with rare proper nouns**: HyDE may generate plausible-but-
  wrong specifics that drag the average embedding off-topic.

## Cost

- +1 chat call per retrieval for N=1 (default).
- +N parallel chat calls for N>1 (wall time ≈ one round-trip thanks to
  `asyncio.gather`).
- On Qwen-2.5-14B / Gemma-4 at FP8, one call is ~200–500 ms.
- No extra GPU memory — the chat model is already loaded.

## Config

| Env var | Default | Meaning |
|---------|---------|---------|
| `RAG_HYDE` | `0` | Enable HyDE. Only `"1"` activates; any other value is off. |
| `RAG_HYDE_N` | `1` | Number of hypothetical excerpts to generate per query. |
| `HYDE_MODEL` | _(falls back to `CHAT_MODEL`)_ | Override the model name sent to the chat endpoint. |

The chat endpoint URL is read from `OPENAI_API_BASE_URL`
(e.g. `http://vllm-chat:8000/v1`) and the API key from `OPENAI_API_KEY`
— the same env vars the rest of the bridge uses.

### Per-KB override (recommended)

P3.0 added per-KB `rag_config` JSONB. Flip HyDE on for a specific KB
without touching the global flag:

```bash
curl -X PATCH http://orgchat.local/api/kb/42/config \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"hyde": true, "hyde_n": 3}'
```

The request-scope flag overlay picks this up and sets `RAG_HYDE=1` +
`RAG_HYDE_N=3` for any chat request that selected KB 42. Other KBs in
the same chat see the merged config (strictest-wins: if ANY selected KB
has `hyde: true`, HyDE runs).

### Tuning guidance

- **N=1** — fast path, +200–500 ms. Use by default.
- **N=3** — +5–10 pp recall on abstract queries, still fits in one
  round-trip via parallel `asyncio.gather`. Recommended for year-long
  KBs.
- **N=5 to 8** — matches the original paper; diminishing returns above
  N=3 on most benchmarks.

## Interaction with other flags

- **`RAG_DISABLE_REWRITE`** — independent. The query-rewriter runs
  first (turning "what about pricing?" into "Acme CRM pricing"), then
  HyDE takes the standardized query as input. Both can be on
  simultaneously, and they compose well: rewriter normalizes phrasing,
  HyDE turns it into a synthetic answer.
- **`RAG_HYBRID`** — independent. HyDE only changes the *query vector*;
  the downstream retrieval (dense-only vs. hybrid RRF) runs unchanged.
- **`RAG_SEMCACHE`** — interacts. The semantic cache is keyed by the
  query vector, so HyDE-on and HyDE-off produce different cache keys.
  This is correct behaviour: a query whose hypothetical-doc drifted in
  one session should not serve stale results to a later identical
  query.

## Fail-open

HyDE is an optimization, never a hard dependency. On any chat-model
failure (timeout, 5xx, malformed JSON, empty response, >2 KB response)
`hyde_embed` returns `None` and the retriever falls through to
embedding the raw query. No request ever fails because of HyDE.

## Eval

Expected movement on our internal eval set (`tests/eval/golden.jsonl`):

| Query shape | HyDE off | HyDE N=1 | HyDE N=3 |
|-------------|---------:|---------:|---------:|
| Abstract / definitional | 55 | +8 pp | +13 pp |
| Keyword / entity-lookup | 82 | +0 pp | +0 pp |
| History-dependent | 68 | +2 pp | +3 pp |

Numbers are chunk-recall@10 on abstract queries — the primary pain point.
Keyword queries should see no movement (HyDE is neutral in the best case
and a latency regression in the worst).

## Implementation notes

- `ext/services/hyde.py` owns generation + averaging. Self-contained,
  no dependencies on the rest of the pipeline.
- `ext/services/retriever.py` lazy-imports the module behind the flag
  check so the default path (HyDE off) never loads the module.
- Unit tests in `tests/unit/test_hyde.py` use `httpx.MockTransport` —
  no network required. Retriever-integration tests in
  `tests/unit/test_retriever_hyde.py` verify the default-off path is
  byte-identical (module is not in `sys.modules`) and that per-KB
  overlay correctly enables the feature.
